Key Concepts

The Programmable Network is based on the core concepts of an Endpoint and a Link. At a simplistic level, a Link connects two Endpoints together (via VPorts), allowing traffic to flow between them (according to one or more Contracts).

Endpoints

There are several types of Endpoint, but they all have the following characteristics:

  • one or more VPorts
  • connect via Links to other Endpoints
  • are located in a Data Centre

Current endpoint types:

  • Physical Ethernet Port
  • VLAN on Physical Ethernet Port
  • Internet (sometimes referred to as DIA or Direct Internet Access)
  • Exchange Partner
  • Virtual Network Function (VNF)

APIs which relate to Endpoints include:

  • GET /1.0.0/inventory/endpoints/customeruuid/customeruuid – Get list of endpoints for a customer
  • GET /1.0.0/inventory/endpoint/endpointuuid – Get information about the specified endpoint
  • POST /1.0.0/inventory/diaendpoint – Create DIA Endpoint
  • POST /1.0.0/inventory/vnfendpoint – Create VNF Endpoint
  • POST /1.0.0/inventory/regularendpoint – Create Physical (Port) Endpoint
  • GET /1.0.0/inventory/endpointname/endpointuuid/endpointuuid – Update Endpoint name

See Interface Types for a list of interface types fore use with Physical Ethernet Ports.

Links

A Link logically connects two Endpoints, allowing traffic to flow between them (once there is a valide Contract – see below).

Links are sometimes also called “Flows”.

APIs which relate to Links include:

  • POST /1.0.0/inventory/link – Create Link
  • GET /1.0.0/inventory/links/customer/customerid – Get active Links
  • GET /1.0.0/inventory/links/history/linkid – Get Link history

See Link/Contract Status Codes for a list of status codes associated with Links.

Contracts

Contracts describe the commercial terms associated with a link and determine the characteristics of the traffic flowing across the link.

Contracts specify:

  • Bandwidth allocated
  • Latency characteristics (Low, Standard, Best Effort)
  • Length of time until the contract expires
  • Behaviour at expiry (Auto Disconnect, Auto Renew, Renew at Hourly Rate)
  • Billing account to which the contract will be charged
  • Price for the contract

APIs which relate to Contracts include:

  • GET /1.0.0/inventory/links/linkid/contract/contractid – Get active Contract by ContractID
  • PUT /1.0.0/inventory/links/linkid/contract/contractid – Update active Contract by ContractID
  • POST /1.0.0/inventory/links/linkid/contract – Create new Contract by ContractID

See Link/Contract Status Codes for a list of status codes associated with Contracts, Latency Codes for a list of codes describing latency options for Contracts, and Renewal Option Codes for a list of codes describing renewal options for Contracts.

Virtual Ports (VPorts)

Endpoints have one or more VPorts which are the “connection points” on an endpoint. For virtual network functions, the VPort represents virtual interfaces. For physical endpoints (Ethernet ports), the VPort represents a VLAN.

APIs which relate to VPorts include:

  • PUT /1.0.0/inventory/endpointname/endpointuuid/endpointuuid – Update VPort name
  • POST /1.0.0/inventory/vnf/vport – Create VNF VPort
  • POST /1.0.0/inventory/vnf/multivport – Create VPort for VNF-VNF Link
  • POST /1.0.0/inventory/regularvport – Create VPort representing a VLAN on a Physical Ethernet Port
  • GET /1.0.0/inventory/vport/vportuuid – Get information about the specified VPort

Virtual Network Functions (Marketplace)

The Programmable Network Marketplace feature provides a mechanism to use virtual network appliances that Telstra or appliance vendors have made available.

Once created, Virtual Network Functions communicate with the Programmable Network via virtual interfaces, which can be connected to Links via VPorts.

To create a VNF, the image for the function must already be added to “My Images”.

APIs which relate to the Marketplace and Virtual Network Functions include:

  • GET /1.0.0/marketplace/image – List images in the Marketplace
  • GET /1.0.0/marketplace/image/id – Get information about the specified image
  • GET /1.0.0/marketplace/image/my_images/ – List images in “My Images”
  • POST /1.0.0/marketplace/image/id/add_to_my_images/ – Add specified image to “My Images”
  • POST /1.0.0/marketplace/image//remove_from_my_images/ – Remove specified image from “My Images”
  • GET /1.0.0/vnc/vnfuuid/get_console – Get the URL and authorisation token for the VNC console of a VNF
  • POST /vnfds/1.0.0/vnf/vnfuuid/suspend/ – Suspend the specified VNF
  • POST /vnfds/1.0.0/vnf/vnfuuid/resume/ – Resume the specified VNF
  • POST /vnfds/1.0.0/vnf/vnfuuid/reboot/ – Reboot the specified VNF
  • GET /vnfds/1.0.0/vnf/vnfuuid/ – Get information about the specified VNF

See VNF Status Codes for a list of status codes associated with VNFs.

Data Centre

Data centres represent the physical location of an Endpoint in the Programmable Network. In particular, each Physical Port, VNF and Exchange Endpoint is associated with a specific data centre.

APIs which relate to Data Centres includes:

  • GET /1.0.0/inventory/datacenters – List data centres

Note: the Programmable Network API uses the US spelling ‘datacenter’

Global Exchange

The Global Exchange is the mechanism in the Programmable Network to interconnect with external networks and services.

Each account can both initiate outgoing connections to an exchange provider and also publish services to the Global Exchange and become an exchange provider themselves. An account that wishes to list services in the Global Exchange should create a Visit Card, which provides the details to be displayed to other users in the Global Exchange list.

APIs which relate to the Global Exchange include:

Exchange consumers:

  • GET /1.0.0/exchange – List all Exchange providers, with POPs
  • GET /1.0.0/exchange/uuid – Get information about the specified Exchange provider
  • POST /1.0.0/exchange-outgoing – Request a connection to an Exchange provider
  • GET /1.0.0/exchange-outgoing/ – List outgoing Exchange connection requests
  • GET /1.0.0/exchange-outgoing/uuid – Get information about the specified outgoing Exchange connection request
  • PUT /1.0.0/exchange-outgoing/uuid – Update an existing outgoing Exchange connection request
  • POST /1.0.0/exchange-outgoing/uuid/close/ – Terminate an existing outgoing Exchange connection request

Exchange providers:

  • GET /1.0.0/account/profile/exchange – Get the current account’s Exchange profile
  • POST /1.0.0/visitcard – Create the Visit Card
  • GET /1.0.0/visitcard/visitcarduuid – View details of the Visit Card
  • GET /1.0.0/exchange-incoming/ – List incoming Exchange connection requests
  • GET /1.0.0/exchange-incoming/uuid – Get information about the specified incoming Exchange connection request
  • POST /1.0.0/exchange-incoming/uuid/accept/ – Accept an incoming Exchange connection request
  • POST /1.0.0/exchange-incoming/uuid/close/ – Reject an incoming Exchange connection request
  • PUT /1.0.0/exchange-incoming/uuid/ – Update an incoming Exchange connection request

Topology and Tag

A Topology is a subset of an account’s resources (Links, Endpoints, etc.) that are logically grouped together. In the web interface, topologies are represented in a canvas view where various objects can be placed together.

Topologies can be used in various ways, depending on the needs of the user. For example, a user might create one topology for Production networks and a separate one for Lab or Experimentation networks. Or, a user might create a topology for a specific application and a different topology for other applications.

The Topology Tag is the label by which a user (and therefore the APIs) refer to topologies. Currently, each resource can only belong to a single topology.

APIs which relate to Topologies include:

  • POST /ttms/1.0.0/topology_tag – Create a named topology tag
  • PUT /ttms/1.0.0/topology_tag/topology_tag_uuid/ – Update a topology tag’s name and/or description
  • GET /ttms/1.0.0/topology_tag – List all topology tags
  • GET /ttms/1.0.0/topology_tag/topology_tag_uuid/ – Get information about the specified topology tag
  • DELETE /ttms/1.0.0/topology_tag/topology_tag_uuid/ – Delete a topology tag
  • GET /ttms/1.0.0/topology_tag/topology_tag_uuid/objects/ – List all objects (Endpoints, Links, VPorts, etc.) associated with the topology tag.

Customer, User and Account

APIs which relate to Customers, Users and Accounts include:

Authentication:

  • POST /1.0.0/auth/generatetoken – Create an authentication token
  • GET /1.0.0/auth/validatetoken – Validate the authentication token and get information about the user (roles, permissions, etc.)

Customers:

  • POST /is/1.0.0/customers – Create a new customer
  • GET /is/1.0.0/customers – List all customers
  • GET /is/1.0.0/customers/customeruuid – Get details of the specified customer
  • PUT /is/1.0.0/customers/customeruuid – Update details of the specified customer
  • GET /is/1.0.0/customers/customeruuid/childcustomers – List all child customers of the specified customer
  • GET /is/1.0.0/customers/customeruuid/childcustomers/info – List all child customers of the specified customer (tabular view)
  • GET /is/1.0.0/customers/customeruuid/directchildcustomers/info – List all immediate child customers of the specified customer
  • PUT /is/1.0.0/customer/customeruuid/tag/taguuid – Update the specified customer tag
  • GET /is/1.0.0/customer/customeruuid/tag – List all tags associated with the specified customer
  • DELETE /is/1.0.0/customer/customeruuid/tag/taguuid – Delete the specified customer tag
  • GET /is/1.0.0/customer/customeruuid/tag/associations – List all customer tag associations for the specified customer

Accounts:

  • GET /1.0.0/account/customeruuid – Get the account information for the specified customer
  • GET /1.0.0/account/customeruuid/user – List all users associated with the customer

Users:

  • POST /is/1.0.0/customer/customeruuid/user – Create a user
  • PUT /is/1.0.0/customer/customeruuid/user/useruuid – Update an existing user
  • GET /is/1.0.0/customer/customeruuid/user – List users associated with the specified customer
  • GET /is/1.0.0/customer/customeruuid/user/useruuid – Get information about the specified user
  • PUT /is/1.0.0/customer/customeruuid/user/useruuid – Update the specified user’s password
  • PUT /is/1.0.0/customer/customeruuid/user/useruuid/changePassword – Update the specified user’s password
  • POST /is/1.0.0/customer/customeruuid/tag – Create a customer tag
  • GET /is/1.0.0/customer/customeruuid/user/useruuid/roles – List all roles associated with the specified user
  • GET /is/1.0.0/customer/customeruuid/user/useruuid/unlock – Unlocks a user that has previously been locked

Pricing:

  • GET /is/1.0.0/customers/customeruuid/defaultpercentage – Get default percentage for the specified customer
  • GET /is/1.0.0/customers/customeruuid/markuppercentage – Get default percentage for the specified customer

See User Roles for details on how to use user roles.

Statistics

APIs which relate to Statistics include:

  • GET /1.0.0/inventory/links-stats/endpoint/endpointuuid/startdate/enddate – Get statistics related to the specified endpoint
  • GET /1.0.0/inventory/links-stats/flow/linkuuid/startdate/enddate – Get statistics related to the specified flow
  • GET /1.0.0/inventory/links-stats/endpointstate/endpointuuid/startdate/enddate – Get statistics related to the state of the specified endpoint
